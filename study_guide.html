6.1 การจัดสรรทรัพยากรคืออะไร?

🎯 อุปมาที่เข้าใจง่าย: คลับห้องสมุด
จินตนาการห้องสมุดที่มีหนังสือ (ทรัพยากร) จำกัด และมีนักศึกษา (Process) หลายคนต้องการยืม
• นักศึกษา A: ต้องการหนังสือเล่ม 1 และ 2 พร้อมกัน
• นักศึกษา B: ต้องการหนังสือเล่ม 2 และ 3 พร้อมกัน
• ถ้า A ถือเล่ม 2 รอเล่ม 1, B ถือเล่ม 2 รอเล่ม 3 → ทั้งคู่รอกันไม่มีทางออก = Deadlock!
บรรณารักษ์ (OS) ต้องชาญฉลาดในการจัดสรรหนังสือเพื่อป้องกันสถานการณ์นี้

ในระบบคอมพิวเตอร์ มีกระบวนการ (Process) หลายตัวทำงานพร้อมกัน และแต่ละกระบวนการต้องการ ทรัพยากร (Resource) เช่น CPU, RAM, เครื่องพิมพ์, ไฟล์ เพื่อให้ทำงานสำเร็จ ระบบปฏิบัติการ (OS) ทำหน้าที่ จัดสรรทรัพยากร เหล่านี้ให้กระบวนการต่างๆ

ประเภทของทรัพยากร (Resource Type): เช่น CPU, RAM, เครื่องพิมพ์ — แต่ละประเภทอาจมีหลาย instance เช่น RAM 8 GB, เครื่องพิมพ์ 2 เครื่อง

ปัญหาหลัก: ทรัพยากรมีจำกัด แต่มีกระบวนการมากมายต้องการใช้พร้อมกัน → OS ต้องจัดสรรอย่างชาญฉลาด

6.2 Safe State vs Deadlock — หัวใจของบทนี้

เป้าหมายของ OS คือการจัดสรรให้ระบบอยู่ใน Safe State เสมอ

Safe State (สถานะปลอดภัย):
มีลำดับกระบวนการ (Safe Sequence) ที่ทุกกระบวนการสามารถทำงานเสร็จได้ทีละตัว โดยใช้ทรัพยากรที่มีอยู่บวกกับที่กระบวนการก่อนหน้าคืนมา

ตัวอย่าง Safe Sequence: ⟨P1 → P3 → P0 → P2⟩ หมายความว่า:
→ P1 ทำงานเสร็จก่อน คืนทรัพยากรทั้งหมด
→ P3 ได้รับทรัพยากรที่ต้องการ ทำงานเสร็จ คืนกลับ
→ ทำแบบนี้ต่อจนทุกกระบวนการเสร็จ
Deadlock (การหยุดชะงัก):
ทุกกระบวนการรอกันเป็นวงกลม ไม่มีใครสามารถทำงานต่อได้ ระบบหยุดนิ่ง

ตัวอย่าง: P1 รอทรัพยากรที่ P2 ถือ, P2 รอทรัพยากรที่ P1 ถือ → วนเวียนไม่มีทางออก
ความสัมพันธ์ที่สำคัญ:
Safe State → ไม่มี Deadlock แน่นอน
Unsafe State → อาจ เกิด Deadlock (ไม่ใช่ว่า Deadlock แน่ๆ)
Deadlock → อยู่ใน Unsafe State แน่นอน

วิธีหา Safe Sequence (ด้วยมือ)

1
ดู Available ว่ามีทรัพยากรอะไรว่างบ้าง
2
หากระบวนการที่ Need ≤ Available (ใช้ทรัพยากรที่มีได้พอ)
3
กระบวนการนั้นทำงานเสร็จ → คืน Allocation ทั้งหมดเข้า Available
4
ทำซ้ำจนทุกกระบวนการเสร็จ (Safe) หรือไม่มีใครทำได้อีก (Deadlock)

6.3 เงื่อนไข 4 ข้อของ Deadlock

Deadlock จะเกิดขึ้นได้ก็ต่อเมื่อเงื่อนไขทั้ง 4 ข้อนี้เกิดขึ้น พร้อมกัน ทำลายเงื่อนไขใดข้อหนึ่งก็ป้องกัน Deadlock ได้

#เงื่อนไขอธิบายตัวอย่างการทำลาย
1Mutual Exclusionทรัพยากรใช้ได้ครั้งละ 1 กระบวนการเท่านั้นทำให้ทรัพยากรแชร์ได้ (แต่ทำยากมาก)
2Hold and Waitกระบวนการถือทรัพยากรอยู่แล้วยังรอทรัพยากรเพิ่มต้องขอทรัพยากรทั้งหมดตั้งแต่เริ่ม หรือขอเมื่อมือว่างเท่านั้น
3No Preemptionทรัพยากรไม่สามารถถูกดึงคืนโดยบังคับ ต้องรอให้คืนเองOS บังคับดึงทรัพยากรคืนได้
4Circular Waitมีวงกลมของกระบวนการที่รอกัน: P1→P2→P3→P1กำหนดลำดับเลขให้ทรัพยากร ขอได้เฉพาะเลขที่สูงกว่า
⚠ จำให้ขึ้นใจ: ต้องมีครบ 4 ข้อพร้อมกัน ถึงจะเกิด Deadlock ขาดข้อใดข้อหนึ่งก็ไม่เกิด

6.4 Resource Allocation Graph (RAG)

กราฟที่แสดงให้เห็นว่าใครถือทรัพยากรอะไร และใครกำลังรออะไรอยู่ ใช้ตรวจสอบ Deadlock กรณี ทรัพยากรมีแค่ 1 instance ต่อประเภท

สัญลักษณ์:
⬤ วงกลม = กระบวนการ (Process)
▭ สี่เหลี่ยม = ประเภทของทรัพยากร (Resource Type)
• จุดในสี่เหลี่ยม = instance (ตัวอย่าง: มี 2 จุด = 2 instances)
→ กระบวนการ → ทรัพยากร = Request Edge (กำลังรอ)
→ ทรัพยากร → กระบวนการ = Assignment Edge (ถือครองอยู่)

วิธีตรวจสอบ Safe/Deadlock จาก RAG

1
หากระบวนการที่สามารถได้รับทรัพยากรที่รออยู่ทั้งหมดได้ (ดูว่ามี instance ว่างพอ)
2
กระบวนการนั้น "สมมติ" ทำงานเสร็จ → ลบออกจากกราฟ คืนทรัพยากรทั้งหมด
3
ทำซ้ำ หากลบได้จนหมด = Safe, ลบไม่หมด = Deadlock

Wait-For Graph (ใช้เมื่อทุก Resource มี 1 instance)

ลบ Node ของ Resource ออก แล้วลาก P→P โดยตรง ถ้ามี cycle (วงกลม) ในกราฟ → Deadlock

ตัวอย่าง: อ่าน RAG

P1 ถือ R1, รอ R2. P2 ถือ R2, รอ R1.
Wait-for graph: P1 → P2 → P1 = มี cycle → Deadlock!

แต่ถ้า P3 ถือ R1 อีกตัว, รอไม่มีอะไร:
P3 ทำงานเสร็จก่อน คืน R1 → P2 ได้รับ R1 ทำงานเสร็จ → P1 ได้รับ R2 → Safe

ขั้นตอนตรวจสอบ Deadlock: RAG Reduction Algorithm

หลักการ: หา Process ที่สามารถได้รับทรัพยากรที่รออยู่ได้ (Request ≤ Available) → สมมติว่ามันทำงานเสร็จ → คืนทรัพยากรทั้งหมด → ทำซ้ำ
ถ้าลบได้จนหมดทุก Process → Safe (ไม่มี Deadlock)
ถ้ายังเหลือ Process ลบไม่ได้ → Deadlock! (Process ที่ลบไม่ได้คือผู้ติด)
เริ่มต้น: Available = B:0, D:0, F:1
1
สแกนหา Process ที่ทำได้:
• A รอ B=1 → B available=0 → ❌ รอไม่ได้
• C รอ F=1 → F available=1 → ✅ C ทำได้!
• E รอ D=1 → D available=0 → ❌ รอไม่ได้
• G ไม่รออะไร → ✅ G ทำได้!
2
ให้ G เสร็จก่อน (ไม่รออะไร) → G คืน F(1 instance)
Available ใหม่: B:0, D:0, F:2
ให้ C เสร็จ (ได้ F ที่รออยู่) → C คืน B(1) และ F(1)
Available ใหม่: B:1, D:0, F:3
3
สแกนใหม่ — Available = B:1, D:0, F:3
• A รอ B=1 → B available=1 → ✅ A ทำได้!
• E รอ D=1 → D available=0 → ❌ รอไม่ได้
ให้ A เสร็จ → A คืน D(1) → Available: B:2, D:1, F:3
4
สแกนสุดท้าย — Available = B:2, D:1, F:3
• E รอ D=1 → D available=1 → ✅ E ทำได้!
ให้ E เสร็จ → ทุก Process เสร็จหมดแล้ว!
✅ Safe Sequence: G → C → A → E
ทุก Process ทำงานเสร็จได้ → ไม่มี Deadlock!

Wait-For Graph — มองให้ชัดขึ้น

ลบ Resource Node ออกทั้งหมด แล้วลาก P→P โดยตรง: "ถ้า P1 รอ Resource ที่ P2 ถือ → วาด P1→P2"

⚠ ข้อควรระวัง: การดู Cycle ใน Wait-For Graph อย่างเดียวอาจทำให้สับสน!
ต้องตรวจสอบด้วยว่า Request Edge แต่ละเส้น — Resource นั้น มี Free Instance หรือเปล่า?
• A→B: B มี 0 free → A ถูกบล็อกจริง
• C→F: F มี 1 free → C ไม่ถูกบล็อก! (ได้ F ทันที)
• E→D: D มี 0 free → E ถูกบล็อกจริง
→ ถอด C ออกจาก "กลุ่มที่อาจ Deadlock" → วงกลมแตก → ไม่มี Deadlock!

เปรียบเทียบ: ถ้า F มีแค่ 2 instances (ไม่มี Free) → เกิด Deadlock!

กรณี F=2 instances: เริ่มต้น Available = B:0, D:0, F:0 (ทุกอย่างเต็มหมด)

Wait-For Graph: A→C→E→A (Cycle) และ C→G→E→A→C (Cycle ยาวกว่า) → Deadlock แน่นอน!

⚠ กฎ: Deadlock เกิดเมื่อทุก Process ในกลุ่มหนึ่งต่างรอกัน AND ไม่มีใครสามารถเสร็จได้ก่อน (ไม่มี Free Resource ที่พอ)
🎯 สรุปวิธีตรวจ Deadlock จาก RAG:
1. ดูว่า Resource ไหนมี Free Instance บ้าง
2. หา Process ที่ Request ≤ Free Available → Process นั้นเสร็จได้ก่อน
3. ให้มันเสร็จ → คืนทรัพยากรทั้งหมด → อัปเดต Available
4. ทำซ้ำจนลบได้หมด (Safe) หรือไม่มีใครเสร็จได้อีก (Deadlock)
5. Process ที่ลบไม่ได้ = ผู้ติด Deadlock

6.5 Safety Algorithm — หัวใจของ Banker's Algorithm

ใช้ตรวจสอบว่าระบบอยู่ใน Safe State หรือไม่ และหา Safe Sequence เมื่อมีทรัพยากรหลายชนิด หลาย instance

ตัวแปรที่ใช้ในอัลกอริทึม

Work[j] = ทรัพยากรที่ "ว่าง" ณ ขณะนั้น (เริ่มต้นเท่ากับ Available)
Finish[i] = true ถ้ากระบวนการ i ทำงานเสร็จแล้ว (เริ่มต้น = false ทุกตัว)

ขั้นตอน Safety Algorithm

0
เตรียมการ: คำนวณ Need = Max − Allocation ทุกกระบวนการ. ตั้งค่า Work = Available, Finish[i] = false ทุกตัว
1
หากระบวนการ i ที่ตรงตามเงื่อนไข: Finish[i] = false และ Need[i] ≤ Work (ทุก resource type)
2
อัปเดต: Work = Work + Allocation[i], Finish[i] = true, บันทึก i ลงใน Safe Sequence → กลับไปขั้นตอน 1
3
ตรวจผล: ถ้า Finish[i] = true ทุกตัว → Safe State มี Safe Sequence. ถ้ายังมี Finish[i] = false → Deadlock

6.6 Request Algorithm — ตัดสินใจให้หรือปฏิเสธ

ขั้นตอน Request Algorithm

1
ตรวจสอบ Max Claim: ถ้า Request[i] > Need[i] → Error! ขอเกินกว่าที่แจ้งไว้ตั้งแต่แรก
2

3
จำลองการให้: Available = Available − Request[i], Allocation[i] = Allocation[i] + Request[i], Need[i] = Need[i] − Request[i]
4
รัน Safety Algorithm: ถ้า Safe → ให้เลย. ถ้า Unsafe → ยกเลิก (rollback) ปฏิเสธคำขอ

6.8 Recovery — การกู้คืนจาก Deadlock

เมื่อตรวจพบ Deadlock มี 2 วิธีในการแก้ไข:

วิธีที่ 1: Process Termination (ยุติกระบวนการ)

แบบวิธีข้อดีข้อเสีย
ยุติทั้งหมดKill ทุกกระบวนการที่ติด Deadlock ทันทีง่าย รวดเร็วสูญเสียงานที่ทำไปแล้วทั้งหมด
ยุติทีละตัวKill ทีละกระบวนการ → รัน Detection ใหม่ → ถ้าหาย Deadlock หยุดสูญเสียน้อยกว่าใช้เวลานาน (รัน Detection ซ้ำหลายรอบ)

วิธีที่ 2: Resource Preemption (บังคับดึงทรัพยากรคืน)

เลือก "เหยื่อ" (Victim Process) แล้วบังคับดึงทรัพยากรที่ถืออยู่คืน มี 3 ประเด็นที่ต้องพิจารณา:

  • เลือกเหยื่อ: เลือกกระบวนการที่สูญเสียน้อยที่สุด (เช่น ทำงานไปน้อยที่สุด)
  • Rollback: ย้อนกระบวนการเหยื่อไปยัง safe checkpoint
  • Starvation: ต้องมีกลไกป้องกันไม่ให้กระบวนการเดิมถูกเลือกเป็นเหยื่อซ้ำแล้วซ้ำเล่า

HW6 RAG (ข้อ 6, 7): การอ่าน Resource Allocation Graph ▼

1. หา Process ที่ ทุก Request Edge สามารถจัดสรรได้ (instance ว่างพอ)

6.Cheat 📋 Cheat Sheet & ข้อผิดพลาดที่พบบ่อย — บทที่ 6

📋 สูตรหลัก
• Need[i] = Max[i] − Allocation[i]
• Available = Total − ΣAllocation
• Safe State ⟹ ไม่เกิด Deadlock
• Deadlock ⟹ Unsafe (แต่ Unsafe ไม่จำเป็น Deadlock)
⚠ ข้อผิดพลาดที่พบบ่อย
• ลืมคำนวณ Need ก่อน Safety
• Need ≤ Work ต้องผ่าน ทุก resource (AND)
• Detection ใช้ Request matrix ไม่ใช่ Need
• Detection: Finish=true ตั้งแต่แรก ถ้า Alloc=0 ทั้งหมด
• Process termination ≠ Resource preemption (HW ข้อ 14)

7.1 พื้นฐาน: Logical vs Physical Address

เมื่อโปรแกรมทำงาน CPU จะอ้างอิงที่อยู่หน่วยความจำแบบ Logical Address (ที่อยู่ตรรกะ) ซึ่งเริ่มจาก 0 เสมอและมีเฉพาะในมุมมองของกระบวนการนั้น แต่ RAM จริงๆ ใช้ Physical Address (ที่อยู่ทางกายภาพ) ที่ระบุตำแหน่งจริงในชิป

ตัวอย่างเปรียบเทียบ:
โปรแกรม A เริ่มที่ logical 0 → Physical 5000
โปรแกรม B เริ่มที่ logical 0 → Physical 8000
ทั้งสองใช้ logical 0 เหมือนกัน แต่อยู่คนละที่จริงๆ ใน RAM
MMU (Memory Management Unit) คือชิปฮาร์ดแวร์ที่แปลง Logical → Physical แบบ Real-time ทุกครั้งที่ CPU เข้าถึงหน่วยความจำ

โครงสร้างกระบวนการในหน่วยความจำ

กระบวนการหนึ่งๆ ประกอบด้วย "word" หลายๆ ตัว แต่ละ word มี logical address ของตัวเอง (0, 1, 2, …, Max−1)

Physical Address = Base + Logical Address (กรณี Contiguous)

7.2 Swapping — สลับกระบวนการเข้าออก RAM

เมื่อ RAM เต็ม OS จะย้ายกระบวนการที่ "รอ" หรือ "ไม่ active" ไปเก็บไว้ที่ฮาร์ดดิสก์ชั่วคราว เรียกว่า Swap Out และดึงกลับมาเมื่อต้องการเรียก Swap In

ปัญหาของ Swapping แบบเต็มกระบวนการ:
ถ้ากระบวนการขนาด 100 MB ต้อง swap ที่ความเร็ว 50 MB/s → ใช้เวลา 2 วินาที แต่กระบวนการทั่วไปรันได้แค่ไม่กี่ ms
→ เสียเวลา Swap มากกว่าเวลาทำงานจริง = ไม่คุ้ม
ระบบทันสมัยจึงใช้ Virtual Memory (Demand Paging) แทน

7.3 Contiguous Memory Allocation — เก็บกระบวนการแบบต่อเนื่อง

🎯 อุปมา: ที่จอดรถในลานจอด
RAM เปรียบเหมือนลานจอดรถ กระบวนการเหมือนรถที่มีขนาดต่างกัน
• Fixed Partition: ตีเส้นช่องจอดขนาดคงที่ล่วงหน้า → รถเล็กจอดช่องใหญ่เสียพื้นที่ (Internal Fragmentation)
• Variable Partition: ไม่ตีเส้นล่วงหน้า ให้รถจอดตามขนาดจริง → พอรถออกไปเกิดช่องว่างกระจาย (External Fragmentation)

วิธีที่ง่ายที่สุด: กระบวนการทั้งหมดถูกเก็บไว้ในบล็อกหน่วยความจำที่ ต่อเนื่องกัน ไม่มีการแตกเป็นชิ้น

วิธีที่ 1: Fixed-Size Partition (แบ่งพาร์ทิชันขนาดคงที่)

RAM ถูกแบ่งเป็นส่วนๆ ล่วงหน้า แต่ละส่วน (Partition) เก็บได้แค่ 1 กระบวนการ

ปัญหา: Internal Fragmentation (การแตกกระจายภายใน)
Partition ขนาด 10 MB แต่กระบวนการมีขนาดแค่ 7 MB → เสีย 3 MB ทิ้งไปโดยเปล่าประโยชน์ "ภายใน" Partition นั้น
Internal Fragmentation = ขนาด Partition − ขนาดกระบวนการ
ตัวอย่าง: Internal Fragmentation

Partition = 5 MB. กระบวนการมีขนาด 3 MB.
Internal Fragmentation = 5 − 3 = 2 MB

วิธีที่ 2: Variable-Size Partition (แบ่งพาร์ทิชันแบบยืดหยุ่น)

ไม่แบ่ง Partition ล่วงหน้า แต่จัดสรรพื้นที่ให้กระบวนการตามขนาดจริง เมื่อกระบวนการออกจากระบบ จะเกิด "รู" (Hole) ว่างขึ้น

ปัญหา: External Fragmentation (การแตกกระจายภายนอก)
มีหน่วยความจำว่างรวมกันพอ แต่กระจายอยู่หลายที่ ไม่มีก้อนเดียวที่ใหญ่พอสำหรับกระบวนการใหม่
ตัวอย่าง: External Fragmentation

RAM ว่าง: [10MB] [5MB] [8MB] กระจายอยู่ 3 ที่ รวม 23 MB
กระบวนการใหม่ต้องการ 20 MB ต่อเนื่อง → ใส่ไม่ได้! แม้มีพื้นที่รวมพอ

วิธีแก้: Compaction (รวมรูว่างเข้าหากัน) หรือใช้ Paging/Segmentation

7.4 Segmentation — แบ่งกระบวนการตามโครงสร้างตรรกะ

กระบวนการถูกแบ่งออกเป็น Segment หลายๆ ส่วนตามความหมายทางโปรแกรม เช่น Code Segment, Data Segment, Stack, Heap แต่ละ Segment มีขนาดต่างกัน และถูกเก็บไว้คนละที่ใน RAM ได้

Segment Table — ตารางที่เก็บข้อมูลของแต่ละ Segment:
• Base = Physical Address เริ่มต้นของ Segment นั้น
• Limit = ขนาด (จำนวน word) ของ Segment นั้น

การแปลง Logical Address → Physical Address (Segmentation)

Physical Address = SegTable[s].base + d
เงื่อนไข: d < SegTable[s].limit (มิฉะนั้น → Segmentation Fault!)

7.5 Paging — แบ่งกระบวนการเป็นหน้าขนาดเท่ากัน

Paging แก้ปัญหา External Fragmentation ด้วยการแบ่งทั้งกระบวนการและ RAM เป็นบล็อกขนาดเท่ากัน

Page = บล็อกของ Logical Address Space (ฝั่งกระบวนการ)
Frame = บล็อกของ Physical Memory ฝั่ง RAM
Page Size = Frame Size เสมอ
แต่ละ Page สามารถอยู่ใน Frame ใดก็ได้ → ไม่ต้องอยู่ติดกัน → ไม่มี External Fragmentation!

Page Table — ตารางแปลงหน้าเป็นเฟรม

แต่ละกระบวนการมี Page Table ของตัวเอง เก็บว่า Page ที่ i อยู่ใน Frame ที่เท่าไร

ฟิลด์ใน Page Tableความหมาย
Frame Numberเฟรมที่เก็บ Page นี้
Valid bit = 1Page อยู่ใน RAM
Valid bit = 0Page อยู่บน Disk (ใช้ใน Virtual Memory)
Dirty bit = 1Page ถูกแก้ไขแล้ว ต้อง Write back ก่อน Swap out

การแปลง Logical Address → Physical Address (Paging)

Logical Address แสดงด้วยคู่ (p, d):

  • p = Page Number = logical_address ÷ page_size (เอาส่วนเต็ม)
  • d = Offset = logical_address mod page_size
Physical Address = PageTable[p] × PageSize + d
(PageTable[p] = Frame Number ของ Page p)
ตัวอย่าง: แปลง Logical → Physical (แบบทศนิยม)

Page size = 4 words. Page Table: [5, 2, 8, 0, 3]

Logical = 13:
p = 13 ÷ 4 = 3 (page 3), d = 13 mod 4 = 1 (offset 1)
Frame = PageTable[3] = 0
Physical = 0 × 4 + 1 = 1

Logical = 9:
p = 9 ÷ 4 = 2, d = 9 mod 4 = 1
Frame = PageTable[2] = 8
Physical = 8 × 4 + 1 = 33

Address Space Size — คำนวณขนาด

สูตรสำคัญ:
ถ้า Logical Address = n bits → Logical Address Space = 2^n
ถ้า Offset = d bits → Page Size = 2^d
ถ้า Physical Address = m bits → Physical Address Space = 2^m
จำนวน Page ทั้งหมด = Logical Address Space ÷ Page Size = 2^(n−d)
จำนวน Frame ทั้งหมด = Physical Address Space ÷ Page Size = 2^(m−d)
ตัวอย่าง: ระบบ 32-bit Logical, 30-bit Physical, Page size 4KB

4KB = 4096 = 2^12 → offset = 12 bits
จำนวน Page = 2^(32−12) = 2^20 = 1,048,576 pages (~1 ล้าน pages)
จำนวน Frame = 2^(30−12) = 2^18 = 262,144 frames (~262K frames)
Page Table ต้องมี 2^20 entries

Internal Fragmentation — คำนวณทีละขั้น

ตัวอย่าง: กระบวนการขนาด 200 words, Page size = 64 words

พื้นที่ที่ได้รับ = 4 × 64 = 256 words
Internal Fragmentation = 256 − 200 = 56 words
(Page สุดท้าย page 4 ว่าง 56 words)

อีกวิธี: 200 mod 64 = 200 − 3×64 = 200 − 192 = 8 (= words ที่ใช้ใน Page สุดท้าย)
Internal Frag = 64 − 8 = 56 words ✓

8.1 Virtual Memory คืออะไร?

โดยปกติกระบวนการทั้งหมดต้องอยู่ใน RAM ก่อนจึงจะทำงานได้ แต่ Virtual Memory ทำให้กระบวนการที่มีขนาดใหญ่กว่า RAM สามารถทำงานได้ โดยเก็บเฉพาะ Page ที่กำลังใช้งาน ไว้ใน RAM ส่วนที่เหลือเก็บบน Disk

ข้อดี: รัน Process ได้มากขึ้น (Degree of Multiprogramming สูงขึ้น) แม้ RAM มีจำกัด
ข้อเสีย: เกิด Page Fault → ต้องดึง Page จาก Disk → ช้ากว่าปกติ
Page Table กับ Virtual Memory:
Valid bit = 1 → Page อยู่ใน RAM (ใช้งานได้ทันที)
Valid bit = 0 → Page อยู่บน Disk → เกิด Page Fault ต้องดึงขึ้นมาก่อน

8.4 Thrashing — ระบบติดขัดจาก Page Fault ถี่เกินไป

อุปมา Thrashing:
เหมือนทำการบ้านบนโต๊ะที่วางหนังสือได้แค่ 2 เล่ม แต่ต้องใช้ถึง 5 เล่ม
→ เปิดเล่มที่ 1 ต้องเอาเล่มอื่นลงชั้น → เปิดเล่มที่ 2 ต้องเอาลงอีก → เสียเวลาแค่ ขนหนังสือ ไม่ได้อ่านจริงเลย
ระบบก็เช่นกัน: Process มี Frame น้อยเกินไป → ต้อง Swap page เข้าออกตลอด → CPU ไม่ได้ประมวลผลจริงเลย

วงจรอุบาทว์ของ Thrashing

เพิ่ม Process (Multiprogramming ↑) → Frame ต่อ Process ลดลง → Page Fault บ่อยมาก → CPU รอ disk ตลอด (Utilization ↓) → OS คิดว่า CPU ว่าง → เพิ่ม Process อีก ← วนซ้ำแย่ลงเรื่อยๆ!

8.5 Frame Allocation — จัดสรร Frame ให้กระบวนการต่างๆ

วิธีที่ 1: Equal Allocation (แจกเท่ากัน)

Frame ต่อกระบวนการ = ⌊Total Free Frames ÷ จำนวนกระบวนการ⌋
ตัวอย่าง: Equal Allocation

Free Frames = 93, กระบวนการ = 5 ตัว
แต่ละกระบวนการได้ = ⌊93÷5⌋ = 18 frames
Frame ที่เหลือสำหรับ Page Fault = 93 − (5×18) = 3 frames

วิธีที่ 2: Proportional Allocation (แจกตามสัดส่วนขนาด)

Frame_i = ⌊(size_i ÷ total_size) × total_frames⌋
ตัวอย่าง: Proportional Allocation

P1 มีขนาด 10 pages, P2 มีขนาด 127 pages. Free Frames = 62
Total size = 10+127 = 137
P1: ⌊(10/137) × 62⌋ = ⌊4.53⌋ = 4 frames
P2: ⌊(127/137) × 62⌋ = ⌊57.47⌋ = 57 frames

9.2 Directory Structure — โครงสร้างการจัดไฟล์

Directory ใช้จัดระเบียบไฟล์ มี 5 โครงสร้างหลัก:

1. Single-Level Directory — ไดเรกทอรีระดับเดียว ▼

มีไดเรกทอรีเดียวสำหรับทุกไฟล์ของทุก User ในระบบ

ข้อดี: ง่ายที่สุด
ข้อเสีย:
• Naming Problem: User สองคนไม่สามารถมีไฟล์ชื่อเดียวกันได้
• ค้นหาช้า (ต้องสแกนทุกไฟล์)
2. Two-Level Directory — ไดเรกทอรีสองระดับ ▼

มี Master File Directory (MFD) → แต่ละ User มี User File Directory (UFD) ของตัวเอง เหมือนต้นไม้ความสูง 2

ข้อดี: แก้ปัญหาชื่อซ้ำ (แต่ละ User มีพื้นที่ชื่อของตัวเอง)
ข้อเสีย: User แต่ละคนเข้าถึงไฟล์ของคนอื่นไม่ได้ (แยกกันสนิท)
3. Tree-Structured Directory — ไดเรกทอรีแบบต้นไม้ ⭐ ใช้จริงที่สุด ▼

มีระดับย่อยได้ไม่จำกัด User สร้าง subdirectory ของตัวเองได้ทุกระดับ นี่คือโครงสร้างที่ Windows, Linux ใช้

Path Name:
Absolute Path: เริ่มจาก Root → /home/user/Documents/report.pdf
Relative Path: จาก Current Directory → Documents/report.pdf
ข้อดี: จัดระเบียบได้ดี ค้นหาเร็ว แต่ละไฟล์มี Path เฉพาะตัว
4. Acyclic-Graph Directory — ไดเรกทอรีแบบกราฟไม่มีวงกลม ▼

อนุญาตให้ แชร์ไฟล์หรือ Directory ได้ ไฟล์เดียวกันปรากฏในหลายตำแหน่ง (เหมือน Symbolic Link)

ไฟล์ที่แชร์จะมี Path Name มากกว่า 1 เส้นทาง การแก้ไขจะมองเห็นจากทุกตำแหน่งทันที
ปัญหา: การลบไฟล์ → ต้องใช้ Reference Count (ลบจริงเมื่อไม่มี Link ชี้แล้ว)
5. General Graph Directory — ไดเรกทอรีแบบกราฟทั่วไป ▼

อนุญาตให้มี Cycle ในโครงสร้าง Directory

ปัญหา: การค้นหาอาจวนลูปไม่สิ้นสุด ต้องกำหนดขอบเขตการ Traverse หรือใช้ Garbage Collection

9.4 วิธีเก็บไฟล์บน Disk (File Allocation Methods)

วิธีที่ 1: Contiguous Allocation (เก็บต่อเนื่อง)

Block ทุก Block ของไฟล์อยู่ติดกันบน Disk Directory เก็บแค่ Block เริ่มต้น และ ขนาด

ข้อดี: เข้าถึงแบบ Sequential และ Random ได้เร็วมาก
ข้อเสีย: External Fragmentation, ไฟล์ขยายขนาดลำบาก

วิธีที่ 2: Linked List Allocation (เก็บแบบเชื่อมโยง)

แต่ละ Block มี Pointer ชี้ไป Block ถัดไป Block ต่างๆ กระจายอยู่ทั่ว Disk

ข้อดี: ไม่มี External Fragmentation, ขยายไฟล์ได้ง่าย
ข้อเสีย: Random Access ช้า (ต้องไล่ตาม Pointer), Pointer กินเนื้อที่, Pointer เสีย → ข้อมูลสูญ

วิธีที่ 3: FAT (File Allocation Table)

ย้าย Pointer ทั้งหมดมาเก็บรวมกันใน Table ตอนต้น Disk แทนที่จะเก็บใน Block Directory เก็บแค่ Block แรกของไฟล์

FAT Entry: แต่ละ Entry บอกว่า Block ถัดไปอยู่ที่ Block ไหน (หรือ EOF ถ้าเป็น Block สุดท้าย)
ข้อดี: Random Access เร็วกว่า Linked (Cache FAT ไว้ใน RAM ได้)

วิธีที่ 4: Indexed Allocation (i-node)

มี Index Block พิเศษ (i-node) ที่เก็บ Pointer ชี้ไปทุก Data Block ของไฟล์

สำหรับไฟล์ขนาดใหญ่มาก ใช้ Indirect Pointer:
• Single Indirect: i-node → Block ของ Pointer → Data
• Double Indirect: i-node → Block ของ Block ของ Pointer → Data
• Triple Indirect: ลึก 3 ระดับ
ข้อดี: Random Access เร็ว, ไม่มี External Fragmentation, ขยายได้ง่าย

ตารางเปรียบเทียบ 4 วิธี

9.4b ตัวอย่างการคำนวณ File Allocation Methods

Contiguous Allocation — คำนวณตำแหน่ง Block

Directory Entry: (start_block, length)
Block ที่ i ของไฟล์ = start_block + i (เริ่มนับจาก 0)
ไฟล์ขนาด n blocks → ใช้ start_block ถึง start_block + n − 1
ตัวอย่าง: ไฟล์ "report" เริ่มที่ Block 14, ขนาด 5 blocks

Block ของไฟล์ = [14, 15, 16, 17, 18]
เข้าถึง Block ที่ 3 (0-indexed) = 14 + 3 = Block 17
เข้าถึง Block ที่ 0 = 14 + 0 = Block 14 (Sequential เริ่มต้น)

Linked List Allocation — ไล่ตาม Pointer

ตัวอย่าง: ไฟล์เริ่มที่ Block 9 (Linked)

Block 9 → ข้อมูล + Pointer ชี้ Block 16
Block 16 → ข้อมูล + Pointer ชี้ Block 1
Block 1 → ข้อมูล + Pointer ชี้ Block 10
Block 10 → ข้อมูล + Pointer = EOF
เข้าถึง Block ที่ 3 (ท้ายสุด): ต้องไล่ 9→16→1→10 = 4 I/O operations!
(ช้ากว่า Contiguous มากสำหรับ Random Access)

Indexed (i-node) — คำนวณ Indirect Pointer

i-node Direct vs Indirect:
Direct Pointer: ชี้ตรงไป Data Block
Single Indirect: ชี้ไป Index Block → Index Block ชี้ไป Data Block
Double Indirect: ชี้ไป Block ของ Index Block → Index Block → Data Block

ถ้า Block size = B bytes, Pointer size = P bytes → แต่ละ Indirect Block มี B/P pointers
ตัวอย่าง: Block size = 512 bytes, Pointer = 4 bytes

Index Block ละ 512/4 = 128 pointers
Direct blocks = 10 (สมมติ i-node มี 10 direct pointers)
Single Indirect: 128 blocks
Double Indirect: 128 × 128 = 16,384 blocks
Triple Indirect: 128^3 = 2,097,152 blocks

ขนาดไฟล์สูงสุด:
(10 + 128 + 16,384 + 2,097,152) × 512 bytes = ~1 GB

จงตอบ: (a) เป็นไฟล์ประเภทใด (b) Octal เลขเป็นเท่าใด (c) Group ทำอะไรได้บ้าง

วิธีคิด:
ตัวที่ 1 = '-' → ไฟล์ธรรมดา
Owner: rwx = 4+2+1 = 7
Group: r-x = 4+0+1 = 5
Others: --x = 0+0+1 = 1

คำตอบสุดท้าย:
(a) ไฟล์ธรรมดา
(b) 751
(c) Group: อ่านและรันได้ (เขียนไม่ได้)
ข้อ 12) File Allocation — Indexed Max Size บทที่ 9

Block size = 4 KB, Pointer size = 4 bytes
Indexed Allocation แบบ single-level — Index Block เก็บ Pointer ได้กี่ตัว และไฟล์ใหญ่สุดเท่าไหร่?

วิธีคิด:
Index Block 1 block = 4096 bytes
จำนวน Pointer ที่เก็บได้ = 4096 / 4 = 1024 pointers
ไฟล์ใหญ่สุด = 1024 × 4 KB = 4,096 KB = 4 MB

คำตอบสุดท้าย: 1024 pointers, ไฟล์สูงสุด 4 MB
